# ADR-003: Supabase for auth, RLS, and storage

**Status:** Accepted  
**Date:** 2026-04-10

## Context

The system needs:
- User authentication (sign-up, login, magic link, session management)
- Row-level access control so users can only see their own data
- File storage for export artifacts (markdown, JSON, HTML, CSV)

Alternatives:

1. **Custom auth (FastAPI + bcrypt + JWT)** — full control, but requires building email verification, magic link, password reset, session revocation from scratch. High security surface area for an MVP.
2. **Auth0 / Clerk** — managed auth services. Good DX, but adds a paid vendor dependency and doesn't include storage.
3. **Supabase** — managed Postgres + Auth + Storage in one platform. RLS policies live in the DB next to the data. Pre-built client libraries for React.
4. **Firebase Auth + Cloud Storage** — Google ecosystem. Non-portable (proprietary SDK). Less SQL-native.

## Decision

Use **Supabase** for auth, RLS, and storage.

- **Auth**: Supabase handles JWT issuance, magic links, password reset. Backend verifies JWTs with `SUPABASE_JWT_SECRET` using `python-jose` — no Supabase client on the backend, just standard JWT decode.
- **RLS**: All 8 tables have `ENABLE ROW LEVEL SECURITY`. Policies use `auth.uid() = user_id`. Service role (used by backend API + worker) bypasses RLS via `SUPABASE_SERVICE_ROLE_KEY`.
- **Storage**: Private `exports` bucket. Objects stored at `{user_id}/{run_id}/{format}.{ext}`. Pre-signed URLs generated by backend for download.

## Consequences

**Positive:**
- Zero auth infrastructure to build or maintain
- RLS enforced at DB level — even a buggy API endpoint can't leak cross-user data
- Magic link + email auth out of the box
- Supabase Storage saves building an S3 integration for MVP
- Supabase Realtime available for future live updates (no extra infra)

**Negative:**
- Supabase free tier has storage and bandwidth limits
- JWT `verify_aud=False` required because Supabase sets `aud: authenticated` not a URL
- Supabase local dev requires `supabase start` (Docker)
- Migrating off Supabase Auth would require re-issuing all user tokens

**Mitigation:**
- Service role key stored in Render environment only — never in frontend or git
- `verify_aud=False` is documented and intentional — Supabase's standard pattern
- Migrations in `supabase/migrations/` are standard SQL — portable to any Postgres
