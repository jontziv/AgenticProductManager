# Supabase Schema & RLS

## Tables

| Table              | PK   | user_id FK | RLS enabled | Notes                          |
|--------------------|------|------------|-------------|--------------------------------|
| user_profiles      | UUID | id (= PK)  | YES         | Mirrors auth.users             |
| intake_runs        | UUID | user_id    | YES         | One per submitted idea         |
| artifacts          | UUID | user_id    | YES         | Versioned; use DISTINCT ON     |
| approvals          | UUID | user_id    | YES         | UNIQUE(run_id)                 |
| qa_reports         | UUID | user_id    | YES         | UNIQUE(run_id)                 |
| queued_jobs        | UUID | user_id    | YES         | Users SELECT only; service writes |
| export_records     | UUID | user_id    | YES         | Links to Storage bucket        |
| stale_markers      | UUID | (via run)  | YES         | Housekeeping only              |

## RLS policy pattern

Every user-owned table uses the same four policies:

```sql
-- SELECT
CREATE POLICY "table_select_own" ON public.table_name
    FOR SELECT USING (auth.uid() = user_id);

-- INSERT
CREATE POLICY "table_insert_own" ON public.table_name
    FOR INSERT WITH CHECK (auth.uid() = user_id);

-- UPDATE
CREATE POLICY "table_update_own" ON public.table_name
    FOR UPDATE USING (auth.uid() = user_id)
    WITH CHECK (auth.uid() = user_id);

-- DELETE (only for tables that allow user deletion)
CREATE POLICY "table_delete_own" ON public.table_name
    FOR DELETE USING (auth.uid() = user_id);
```

**Service role bypasses RLS** — the backend API always uses `SUPABASE_SERVICE_ROLE_KEY` and connects via `asyncpg` directly, so RLS policies don't restrict server-side writes.

## Artifact versioning

Artifacts are append-only (never UPDATE). Each regeneration inserts a new row with `version + 1`.

To read the latest version per type:

```sql
SELECT DISTINCT ON (artifact_type) *
FROM artifacts
WHERE run_id = $1
ORDER BY artifact_type, version DESC;
```

This is implemented in `app/db/queries.py:ArtifactsDB.list_by_run()`.

## Job queue claim (worker)

```sql
SELECT * FROM queued_jobs
WHERE status = 'queued'
ORDER BY priority ASC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

`SKIP LOCKED` prevents two workers from claiming the same job. Implemented in `app/db/queries.py:JobsDB.claim_next()`.

## Storage bucket

Bucket: `exports` (private)
Object path: `{user_id}/{run_id}/{format}.{ext}`

Service role writes objects. Users access via pre-signed URLs (generated by backend, 1-hour TTL).

## Migration conventions

1. Files in `supabase/migrations/` are numbered `NNN_description.sql`
2. Run with: `supabase db push` (local) or apply in Supabase Dashboard (production)
3. Never edit an applied migration — add a new one
4. Test migrations: `supabase db reset` re-runs from scratch

## Adding a new table

```sql
-- 1. Create table with uuid PK, user_id FK, timestamps
CREATE TABLE public.new_table (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id     UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    -- your columns
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 2. Enable RLS
ALTER TABLE public.new_table ENABLE ROW LEVEL SECURITY;

-- 3. Add policies (see pattern above)

-- 4. Add updated_at trigger
CREATE TRIGGER trg_new_table_updated_at
    BEFORE UPDATE ON public.new_table
    FOR EACH ROW EXECUTE FUNCTION public.set_updated_at();

-- 5. Add indexes
CREATE INDEX idx_new_table_user_id ON public.new_table (user_id, created_at DESC);
```

## Common query patterns (asyncpg)

```python
# Insert and return
row = await conn.fetchrow(
    "INSERT INTO intake_runs (user_id, title, raw_input) VALUES ($1, $2, $3) RETURNING *",
    user_id, title, raw_input
)

# Upsert artifact
await conn.execute("""
    INSERT INTO artifacts (run_id, user_id, artifact_type, content, version)
    VALUES ($1, $2, $3, $4, $5)
    ON CONFLICT DO NOTHING
""", run_id, user_id, artifact_type, json.dumps(content), version)
```
